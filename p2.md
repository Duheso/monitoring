**Aplicativo de Navegação 3D para São Paulo**  
Stack: Python (FastAPI) + React (PWA) + PostgreSQL | Self-Hosted | Idioma: PT-BR

---

## 📌 Visão Geral do Produto

NavSP é um Progressive Web App (PWA) mobile-first de navegação turn-by-turn para São Paulo com renderização 3D estilo Tesla/carros chineses. O mapa exibe perspectiva inclinada da pista, sinalização antecipada de manobras, dados de tráfego em tempo real e recalculo automático de rota. O usuário pode salvar locais e rotas favoritas, fazer login via Google/Apple e ouvir instruções de voz opcionais em PT-BR. **Toda a infraestrutura é 100% self-hosted on-premise — sem vendor lock-in, sem limites de uso.**

## 🚦 Estratégia de Tráfego em Tempo Real (Self-Hosted)

O tráfego em tempo real é o ponto mais desafiador do self-hosted. Estratégia em duas fases:

### Fase 4a — FCD (Floating Car Data) dos Próprios Usuários
- Coletar posição GPS + velocidade dos usuários (com consentimento explícito)
- Armazenar no PostgreSQL: `gps_probes (user_id, lat, lng, speed, timestamp)`
- Processar via job Python (a cada 2 min): calcular velocidade média por segmento OSM
- Gerar `traffic.tar` no formato Valhalla → injetar no roteamento
- Colorir mapa com dados de probe: verde/amarelo/vermelho por velocidade vs. freeflow speed

### Fase 4b — Dados OSM de Congestionamento Histórico
- OSM contém `maxspeed` por via → calcular razão velocidade_atual / velocidade_max
- Usar como baseline até ter dados de probe suficientes

> Esta abordagem é 100% self-hosted, sem nenhuma API externa de tráfego.

## 📊 Modelo de Banco de Dados (PostgreSQL + PostGIS)

```sql
-- Extensão geoespacial
CREATE EXTENSION IF NOT EXISTS postgis;
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- Usuários
CREATE TABLE users (
    id          UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    email       TEXT UNIQUE NOT NULL,
    name        TEXT,
    avatar_url  TEXT,
    provider    TEXT NOT NULL CHECK (provider IN ('google', 'apple')),
    provider_id TEXT UNIQUE NOT NULL,
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    updated_at  TIMESTAMPTZ DEFAULT NOW()
);

-- Locais Favoritos (com geometria PostGIS)
CREATE TABLE favorite_places (
    id          UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    name        TEXT NOT NULL,
    address     TEXT,
    location    GEOGRAPHY(POINT, 4326) NOT NULL,  -- PostGIS geometry
    icon        TEXT DEFAULT 'star',
    created_at  TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX idx_favorite_places_location ON favorite_places USING GIST(location);
CREATE INDEX idx_favorite_places_user ON favorite_places(user_id);

-- Rotas Favoritas
CREATE TABLE favorite_routes (
    id              UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    name            TEXT NOT NULL,
    origin_name     TEXT NOT NULL,
    origin_point    GEOGRAPHY(POINT, 4326) NOT NULL,
    dest_name       TEXT NOT NULL,
    dest_point      GEOGRAPHY(POINT, 4326) NOT NULL,
    geometry        GEOGRAPHY(LINESTRING, 4326),  -- geometria da rota calculada
    distance_meters INTEGER,
    duration_secs   INTEGER,
    created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- Probe GPS dos usuários (para tráfego FCD)
CREATE TABLE gps_probes (
    id          BIGSERIAL PRIMARY KEY,
    user_id     UUID REFERENCES users(id) ON DELETE SET NULL,
    location    GEOGRAPHY(POINT, 4326) NOT NULL,
    speed_kmh   REAL,
    heading_deg REAL,
    recorded_at TIMESTAMPTZ NOT NULL
);
CREATE INDEX idx_gps_probes_location ON gps_probes USING GIST(location);
CREATE INDEX idx_gps_probes_time ON gps_probes(recorded_at DESC);
-- Particionamento por dia para performance (opcional em escala)
```

### FASE 1 — Mapa Base + Rota (100% Self-Hosted)

**M1.1 — Preparar dados OSM e tileserver-gl**
- Script `geo-services/tiles/download-sp-mbtiles.sh`:
  - Baixar MBTiles de SP da OpenMapTiles (arquivo `.mbtiles` para uso local)
  - Configurar `tileserver-gl` com estilo `liberty` adaptado para PT-BR
- Configurar endpoint interno: `http://tiles:8080/styles/liberty/style.json`
- **Testes:** `test_tiles.py` (backend) — endpoint `/api/v1/geo/tiles/health` retorna status do tileserver

**M1.2 — Preparar OSRM com dados Brasil/SP**
- Script `geo-services/osrm/download-brazil.sh`:
  - `wget https://download.geofabrik.de/south-america/brazil-latest.osm.pbf`
  - Ou extract apenas SP: `wget .../brazil/sudeste/sao-paulo-latest.osm.pbf`
- Script `geo-services/osrm/preprocess.sh`:
  - `osrm-extract -p /opt/car.lua brazil.osm.pbf`
  - `osrm-partition brazil.osrm`
  - `osrm-customize brazil.osrm`
- OSRM rodando em `http://osrm:5000`
- **Testes:** `test_geo_proxy.py` — mock da chamada ao OSRM, testar formatação do response

**M1.3 — Preparar Photon (geocoding self-hosted)**

  - Configurar container Photon com dados locais
- **Testes:** `test_geo_proxy.py` — mock Photon, testar parse de resultados

**M1.4 — Backend: endpoint de geocoding (proxy Photon)**
- `GET /api/v1/geo/search?q=...&lat=...&lon=...`
- Chama Photon interno, filtra por bbox SP, retorna top 5 resultados
- Response: `[{ name, address, lat, lng, type }]`
- **Testes:** `test_geo_proxy.py`:
  - `test_search_returns_results` — mock Photon, verifica formato
  - `test_search_filters_bbox` — garante resultados fora de SP são removidos

## 🌐 Deploy On-Premise (Self-Hosted)

### Configuração Recomendada (Docker Compose On-Premise)

```
[Servidor 1 — App]           [Servidor 2 — Geo]
FastAPI + PostgreSQL          OSRM + Valhalla
Photon + Nginx                tileserver-gl
16GB RAM / 8vCPU              32GB RAM / 8vCPU
200GB SSD                     300GB SSD
```

**Stack de produção:**
- **Nginx**: reverse proxy + SSL termination + gzip + caching de assets
- **Certbot**: certificado TLS via Let's Encrypt (ou cert próprio on-prem)
- **Logrotate**: rotação de logs dos containers
- **Cron**: atualização mensal dos dados OSM do Geofabrik (`cron/update-osm.sh`)
- **pg_dump**: backup diário automático do PostgreSQL

### Atualização de Dados OSM (Script Mensal)
```bash
# cron/update-osm.sh
wget https://download.geofabrik.de/south-america/brazil/sudeste/sao-paulo-latest.osm.pbf
docker exec osrm osrm-extract ... && osrm-partition ... && osrm-customize ...
docker exec valhalla valhalla_build_tiles ...
docker exec photon photon --reindex ...
docker restart osrm valhalla photon
```

## ⚠️ Riscos e Mitigações

| Risco | Impacto | Mitigação |
|---|---|---|
| OSRM requer 16–32GB RAM para Brasil | Alto | Usar extract só de SP (~4GB PBF, ~8GB RAM) para MVP |
